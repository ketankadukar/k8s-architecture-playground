Kubernetes Overview

Attractive blog‑style notes for learning Kubernetes

Introduction

Kubernetes is a portable, extensible, open source platform for managing containerized workloads and services. It facilitates declarative configuration and automation, and has a large, rapidly growing ecosystem.

The name Kubernetes originates from Greek, meaning helmsman or pilot. “K8s” is an abbreviation formed by counting the eight letters between “K” and “s”. Google open sourced Kubernetes in 2014, combining 15+ years of experience running production workloads with community best practices.

Why You Need Kubernetes

Containers bundle and run applications, but in production you need resilience and automation. Kubernetes provides a framework to run distributed systems reliably, handling scaling, failover, and deployment patterns like canary releases.

Key Features

  • Service discovery & load balancing: Expose containers via DNS/IP and balance traffic.
  • Storage orchestration: Mount local or cloud storage automatically.
  • Automated rollouts & rollbacks: Declaratively manage desired state and updates.
  • Automatic bin packing: Efficiently schedule containers based on CPU/memory needs.
  • Self‑healing: Restart, replace, and hide unhealthy containers until ready.
  • Secrets & config management: Securely store and update sensitive data.
  • Batch execution: Manage CI/CD and batch workloads.
  • Horizontal scaling: Scale apps via command, UI, or metrics.
  • IPv4/IPv6 dual‑stack: Support both address families.
  • Extensibility: Add features without changing upstream code.

What Kubernetes Is Not

Kubernetes is not a traditional monolithic PaaS. It provides building blocks for developer platforms but preserves flexibility.

  • Does not limit application types — supports stateless, stateful, and data‑processing workloads.
  • Does not deploy source code or build applications — CI/CD is external.
  • Does not provide built‑in middleware, databases, or storage systems — you can run or connect them yourself.
  • Does not dictate logging/monitoring — integrations are optional.
  • Does not mandate a configuration language — it exposes a declarative API.
  • Not a simple orchestration system — it continuously drives actual state toward desired state.

Historical Context

Traditional Deployment Era

Applications ran on physical servers without resource boundaries, leading to underutilization and high costs.

Virtualized Deployment Era

Virtualization allowed multiple VMs per server, isolating applications and improving resource utilization.

Container Deployment Era

Containers share the OS kernel, making them lightweight and portable across environments.

Benefits of Containers

  • Agile application creation and deployment.
  • Continuous development, integration, and deployment.
  • Separation of concerns between Dev and Ops.
  • Observability of application health and metrics.
  • Consistency across dev, test, and production.
  • Cloud and OS portability.
  • Application‑centric management.
  • Microservices architecture support.
  • Resource isolation and predictable performance.
  • High efficiency and density.
